# Plugin Creation Rules

These rules govern all plugins generated by the `plugin-create` skill. Apply them silently. Flag conflicts between user requirements and these rules before generating code.

---

## 1. Directory Structure

Every plugin lives at `src/plugins/{plugin-name}/` and follows this layout:

```
src/plugins/{plugin-name}/
├── commands/
│   └── {command-name}/
│       ├── handler.ts      # Command logic
│       ├── input.ts        # Zod input schema
│       ├── output.ts       # Zod output schema + Handlebars template
│       ├── types.ts        # Local types (only if needed for complex params)
│       └── index.ts        # Re-exports: handler, schemas, template
├── __tests__/unit/
│   ├── {command-name}.test.ts
│   └── helpers/
│       ├── mocks.ts
│       └── fixtures.ts
├── hooks/                  # Only if plugin defines its own hooks
│   └── {hook-name}/
│       └── index.ts
├── utils/                  # Only if shared helpers exist across commands
├── manifest.ts             # Plugin definition
├── index.ts                # Public exports
├── schema.ts               # Zod state schema (only if plugin persists data)
└── constants.ts            # Plugin-level constants (only if needed)
```

Rules:

- `types.ts` is optional - only add it if the handler has complex intermediate types (e.g., `NormalisedParams`, `BuildResult`)
- `utils/` is optional - only add it if 2+ commands share the same helper logic
- `hooks/` is optional - only add it if the plugin _defines_ new hooks (not just uses existing ones)
- `schema.ts` is optional - only add it if the plugin persists data to state
- `constants.ts` is optional - only add it if there are plugin-scoped constants used in 2+ files

## 2. Naming Conventions

| Item                    | Convention                              | Example                          |
| ----------------------- | --------------------------------------- | -------------------------------- |
| Plugin directory        | `kebab-case`                            | `token-rewards`                  |
| Plugin name (manifest)  | `kebab-case`                            | `token-rewards`                  |
| Command name (manifest) | `kebab-case`                            | `create`, `list-all`             |
| Manifest export         | `{PascalCase}PluginManifest`            | `TokenRewardsPluginManifest`     |
| Input schema export     | `{PluginName}{CommandName}InputSchema`  | `TokenRewardsCreateInputSchema`  |
| Output schema export    | `{PluginName}{CommandName}OutputSchema` | `TokenRewardsCreateOutputSchema` |
| Handler class           | `{PluginName}{CommandName}Command`      | `TokenRewardsCreateCommand`      |
| Handler function        | `{camelCase}{CommandName}`              | `tokenRewardsCreate`             |
| Template constant       | `{SCREAMING_SNAKE}_TEMPLATE`            | `TOKEN_REWARDS_CREATE_TEMPLATE`  |
| State schema export     | `{PascalCase}DataSchema`                | `TokenRewardsDataSchema`         |

## 3. Handler Patterns

Choose the handler pattern based on the command type:

### Pattern A: Simple Command (no transaction lifecycle)

Use for: read-only commands, config commands, list/view commands.

```typescript
// handler.ts
export class MyPluginListCommand implements Command {
  async execute(args: CommandHandlerArgs): Promise<CommandResult> {
    const validArgs = MyPluginListInputSchema.parse(args.args);
    // logic
    return { result: output };
  }
}

export async function myPluginList(
  args: CommandHandlerArgs,
): Promise<CommandResult> {
  return new MyPluginListCommand().execute(args);
}
```

### Pattern B: Transaction Command (build → sign → execute lifecycle)

Use for: commands that submit Hedera transactions.

```typescript
// handler.ts
export class MyPluginCreateCommand extends BaseTransactionCommand<
  CreateNormalisedParams,
  CreateBuildResult,
  CreateSignResult,
  CreateExecuteResult
> {
  async normalizeParams(args: CommandHandlerArgs): Promise<CreateNormalisedParams> { ... }
  async buildTransaction(args, params): Promise<CreateBuildResult> { ... }
  async signTransaction(args, params, buildResult): Promise<CreateSignResult> { ... }
  async executeTransaction(args, params, buildResult, signResult): Promise<CreateExecuteResult> { ... }
  async outputPreparation(args, params, buildResult, signResult, executeResult): Promise<CommandResult> { ... }
}

export async function myPluginCreate(args: CommandHandlerArgs): Promise<CommandResult> {
  return new MyPluginCreateCommand().execute(args);
}
```

Rules:

- **Always export both** the class AND the function wrapper for backward compatibility
- Use Pattern A by default; only use Pattern B when the command submits a transaction
- `normalizeParams` is where input schema parsing happens — not in `buildTransaction`
- Never put business logic in the function wrapper — it delegates to the class

## 4. Input Schemas

Location: `commands/{name}/input.ts`

```typescript
import { z } from 'zod';
import { EntityReferenceSchema, MemoSchema, KeySchema } from '@/core/schemas';

export const MyPluginCreateInputSchema = z.object({
  name: AliasNameSchema.describe('Name or alias for the resource'),
  // ...
});

export type MyPluginCreateInput = z.infer<typeof MyPluginCreateInputSchema>;
```

Rules:

- Always import from `@/core/schemas` — reuse `EntityIdSchema`, `MemoSchema`, `KeySchema`, `AmountInputSchema`, `AccountReferenceSchema`, `EntityReferenceSchema`, `EmptyOrNullableKeyListSchema`, etc.
- Always export both the schema and the inferred TypeScript type
- Use `.describe()` on every field — it powers CLI help text
- Use `z.superRefine()` for cross-field validation (mutually exclusive options, conditional requirements)
- Nullable/clearable key lists: use `EmptyOrNullableKeyListSchema` (not inline `z.array(KeySchema).or(NullLiteralSchema).optional().default([])`)
- Nullable/clearable memo: use `MemoSchema.or(NullLiteralSchema).optional()`

---

## 5. Output Schemas

Location: `commands/{name}/output.ts`

```typescript
import { z } from 'zod';
import {
  EntityIdSchema,
  NetworkSchema,
  TransactionIdSchema,
} from '@/core/schemas';

export const MyPluginCreateOutputSchema = z.object({
  id: EntityIdSchema,
  network: NetworkSchema,
  transactionId: TransactionIdSchema,
});

export type MyPluginCreateOutput = z.infer<typeof MyPluginCreateOutputSchema>;

export const MY_PLUGIN_CREATE_TEMPLATE = `
✅ Resource created: {{id}}
   Network: {{network}}
   Transaction: {{hashscanLink transactionId "transaction" network}}
`.trim();
```

Rules:

- Always export both the schema and the inferred type
- Always export a Handlebars template constant
- Use `{{hashscanLink entityId "account"|"transaction"|"token" network}}` for explorer links
- Use `{{#if field}}...{{/if}}` for optional fields
- Use `{{#each items}}...{{/each}}` for arrays
- Template success prefix: `✅` for create/update/delete, no prefix for list/view

## 6. Manifest Structure

Location: `manifest.ts`

```typescript
import type { PluginManifest } from '@/core';
import { OptionType } from '@/core/types/shared.types';
import { MyPluginCreateCommand } from './commands/create/handler';
import {
  MyPluginCreateOutputSchema,
  MY_PLUGIN_CREATE_TEMPLATE,
} from './commands/create/output';

export const MyPluginPluginManifest: PluginManifest = {
  name: 'my-plugin',
  version: '1.0.0',
  displayName: 'My Plugin',
  description: 'Short description of what this plugin does',

  commands: [
    {
      name: 'create',
      summary: 'Create a resource',
      description: 'Longer description of the create command',
      options: [
        {
          name: 'name',
          short: 'n',
          type: OptionType.STRING,
          required: true,
          description: 'Name of the resource',
        },
      ],
      command: new MyPluginCreateCommand(),
      output: {
        schema: MyPluginCreateOutputSchema,
        humanTemplate: MY_PLUGIN_CREATE_TEMPLATE,
      },
    },
  ],
};

export default MyPluginPluginManifest;
```

Rules:

- `name` must be unique across all plugins — check existing plugin names before generating
- Every option must have: `name`, `short` (single char), `type`, `required`, `description`
- `short` must not conflict with reserved short options: `h`, `V`, `N`, `P`, try to add them if you can, also if you would use hooks check if hook does not reserve any short options e.g. `batchify` hook reserves `-B` char as an option and `scheduled` reserves `-X`
- `command: new MyPlugin{Name}Command()` for class-based (preferred)
- `handler: myPluginName` as fallback only for simple commands that don't justify a class
- Include `registeredHooks` only if the command opts into cross-plugin hooks (batchify, scheduled, etc.)
- Include `requireConfirmation` for destructive commands (delete, clear)

## 7. State Management

Only add state if the plugin needs to persist data between CLI invocations.

```typescript
// schema.ts
import { z } from 'zod';
import { AliasNameSchema, EntityIdSchema, NetworkSchema } from '@/core/schemas';

export const MyPluginDataSchema = z.object({
  name: AliasNameSchema,
  id: EntityIdSchema,
  network: NetworkSchema,
});

export type MyPluginData = z.infer<typeof MyPluginDataSchema>;
```

Rules:

- Always use `AliasNameSchema` for alias/name fields
- Always include `network: NetworkSchema` — resources are network-scoped
- State is accessed via `args.state` in handlers
- Never store private keys in state

---

## 8. Error Handling

Import errors from `@/core/errors`. Map to these types:

| Situation                               | Error Class                |
| --------------------------------------- | -------------------------- |
| Invalid user input                      | `ValidationError`          |
| Missing config (operator not set, etc.) | `ConfigurationError`       |
| Resource not found in state             | `NotFoundError`            |
| Unexpected internal failures            | `InternalError`            |
| Insufficient balance                    | `InsufficientBalanceError` |

Never throw raw `Error` — always use a typed error from `@/core/errors`.

---

## 9. Hook Registration

To opt a command into cross-plugin hooks, add `registeredHooks` to the command spec in the manifest:

```typescript
registeredHooks: [
  { hook: 'batchify-set-batch-key', phase: 'preSignTransaction' },
  { hook: 'scheduled', phase: 'preSignTransaction' },
  { hook: 'batchify-add-transaction', phase: 'preExecuteTransaction' },
];
```

Rules:

- Only transaction commands (Pattern B) should register for `batchify-*` and `scheduled` hooks
- Never register hooks in a read-only command
- Hooks are declared on the command in the manifest — not in the handler

## 11. Default Plugin Registration

Registration file: `src/core/shared/config/cli-options.ts`

To register as a default plugin, add:

1. An import: `import myPluginManifest from '@/plugins/my-plugin/manifest';`
2. An entry in `DEFAULT_PLUGIN_STATE`: `myPluginManifest,`

Only do this if the user explicitly requested it during the interview.

## 12. Unit Tests

Every plugin must have `__tests__/unit/helpers/mocks.ts` and `__tests__/unit/helpers/fixtures.ts`. Test files import from these helpers — never define test data or mock factories inline in test files.

### mocks.ts

Contains `make*` factory functions that return typed Jest mocks. Always include these three base factories:

```typescript
import type { CoreApi } from '@/core/core-api/core-api.interface';
import type { CommandHandlerArgs } from '@/core/plugins/plugin.interface';
import type { Logger } from '@/core/services/logger/logger-service.interface';
import type { StateService } from '@/core/services/state/state-service.interface';

// Base logger mock — reused across all command mocks
export const makeLogger = (): jest.Mocked<Logger> => ({
  info: jest.fn(),
  error: jest.fn(),
  debug: jest.fn(),
  warn: jest.fn(),
  setLevel: jest.fn(),
});

// Plugin-specific API mock — add only the api sub-services this plugin uses
export const makeApiMock = (
  overrides?: Partial<jest.Mocked<CoreApi>>,
): jest.Mocked<CoreApi> =>
  ({
    // add plugin-specific services here, e.g. api.token, api.account
    ...overrides,
  }) as unknown as jest.Mocked<CoreApi>;

// Generic CommandHandlerArgs factory — used by every command test
export const makeCommandArgs = (params: {
  api: jest.Mocked<CoreApi>;
  logger?: jest.Mocked<Logger>;
  args?: Record<string, unknown>;
}): CommandHandlerArgs => ({
  args: { ...(params.args ?? {}) },
  api: params.api,
  state: {} as unknown as StateService,
  config: params.api.config,
  logger: params.logger ?? makeLogger(),
  hooks: new Map(),
});
```

Beyond the base factories, add one `make{Plugin}{Command}Args` factory per command — pre-filled with valid default args so tests only override what they care about:

```typescript
export const makeMyPluginCreateArgs = (params: {
  api: jest.Mocked<CoreApi>;
  logger?: jest.Mocked<Logger>;
  args?: Partial<{ name: string /* ...command options */ }>;
}): CommandHandlerArgs =>
  makeCommandArgs({
    api: params.api,
    logger: params.logger,
    args: {
      name: 'my-resource', // sensible default
      ...params.args,
    },
  });
```

Rules:

- All mock factories use `jest.fn()` for methods — never real implementations
- Use `as unknown as jest.Mocked<T>` when casting partial mocks to full types
- Support an `overrides` parameter on API-level mocks so individual tests can swap one method without rebuilding the whole mock
- Never import from `fixtures.ts` in `mocks.ts` — the dependency is one-way (fixtures ← mocks, never the reverse)

### fixtures.ts

Contains typed constant data used across test files. Organise into sections:

```typescript
// 1. Primitive ID/key constants — reused in both valid and invalid fixtures
export const mockAccountIds = {
  operator: '0.0.100000',
  treasury: '0.0.123456',
};

// 2. Valid input data — one entry per command's happy path
export const validMyPluginCreateInput = {
  name: 'my-resource',
  // ...all required fields with valid values
};

// 3. Invalid input data — cover validation edge cases
export const invalidMyPluginCreateInputMissingName = {
  // name field omitted — triggers required-field error
};

// 4. State/persistence data — if the plugin has schema.ts
export const mockMyPluginData = {
  basic: { id: '0.0.123456', name: 'my-resource', network: 'testnet' },
};

// 5. Factory functions for complex data — use make* prefix, support overrides
export const makeMyPluginData = (
  overrides: Partial<MyPluginData> = {},
): MyPluginData => ({
  id: '0.0.123456',
  name: 'my-resource',
  network: 'testnet',
  ...overrides,
});

// 6. Pre-configured scenario sets — named by scenario
export const mockListScenarios = {
  empty: [],
  twoItems: [
    makeMyPluginData({ id: '0.0.1111' }),
    makeMyPluginData({ id: '0.0.2222' }),
  ],
};
```

Rules:

- Every `make*` factory must accept an `overrides` parameter of `Partial<T>` and spread it last
- Cover at minimum: one valid object, one invalid object (missing required field), one invalid object (wrong format) per command input
- Pre-built scenario sets (empty, single item, multiple items) go in fixtures — not constructed inline in test files
- Fixture constants are `const` — never mutated; if a test needs a variation, call the factory with overrides instead of modifying a shared constant

### Test files ({command}.test.ts)

Each command gets its own test file. Minimum required coverage per command:

```typescript
import { makeLogger, makeApiMock, makeMyPluginCreateArgs } from './helpers/mocks';
import { validMyPluginCreateInput, invalidMyPluginCreateInputMissingName } from './helpers/fixtures';

describe('MyPlugin create command', () => {
  let api: jest.Mocked<CoreApi>;
  let logger: jest.Mocked<Logger>;

  beforeEach(() => {
    api = makeApiMock();
    logger = makeLogger();
  });

  it('returns expected output on valid input', async () => {
    // arrange
    api.someService.someMethod.mockResolvedValue({ ... });
    const args = makeMyPluginCreateArgs({ api, logger });

    // act
    const result = await myPluginCreate(args);

    // assert
    expect(result.result).toMatchObject({ ... });
  });

  it('throws ValidationError when name is missing', async () => {
    const args = makeMyPluginCreateArgs({ api, logger, args: { name: undefined } });
    await expect(myPluginCreate(args)).rejects.toThrow(ValidationError);
  });
});
```

Rules:

- Use `beforeEach` to reset mocks — never share mock state between tests
- Arrange / Act / Assert structure in every `it` block (comments optional but order mandatory)
- Import test data from fixtures, never hardcode strings inline in test assertions
- One `describe` block per command file, named `'{PluginName} {command} command'`
